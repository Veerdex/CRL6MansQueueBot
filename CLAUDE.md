# CRL 6 Mans — Project Memory

Discord bot + companion website for a small, high-skill (Champ+) Rocket League community running 3v3 6-mans pickup games. This file is the durable source of truth for game mechanics and architecture direction — read it in full before working on any phase of this project.

Source docs (original spec + Q&A) live at `C:\Users\grant\Downloads\CRL 6 Mans Queue Bot\` (`6mans_bot_spec.md`, `Extra Context.txt`, `claude_code_prompt.md`). This file supersedes them where they conflict — see "Resolved clarifications" below.

## Architecture direction

- **Website**: Next.js on **Vercel**.
- **Database**: **Supabase** (Postgres) — shared source of truth for both the website and the Discord bot. Not SQLite (original spec assumed a bot-local SQLite file; superseded).
- **Discord bot**: **not** a separate process — it's a set of Next.js API routes living inside `web/`, deployed as part of the same Vercel project as the website. Decided this session: every bot feature (queue buttons, vote/draft buttons, slash commands) is Discord-interaction-triggered, and Discord supports an **HTTP Interactions Endpoint** model (Discord POSTs each interaction to a webhook URL) as an alternative to a persistent gateway/WebSocket connection — this fits Vercel's serverless model with no extra hosting platform needed. See "Discord bot runtime architecture" below for the concrete mechanics (signature verification, the 3-second response limit, background sweeps via Supabase `pg_cron`).
- **Build approach**: incremental, one piece at a time. Each phase gets a brainstorm/design discussion *before* implementation — never build ahead of an agreed design.

## Game mechanics (resolved design)

### Queueing
- **Universal Queue** (open to all ranks) and **Rank Queue** (open to everyone immediately, regardless of placement — see "Bands / ranks" below) both exist.
- **Universal Queue is fully unranked**: results there never change MMR. Only Rank Queue series run the Elo update. This is a deliberate deviation from the original spec doc (which assumed one MMR shared symmetrically by both queues) — decided this session.
- Universal Queue games still count toward general games-played counters (season eligibility, lifetime placement counter) and still display against a player's current MMR for context — they just never move it.
- A player may sit in both queues simultaneously. If one pops (6/6), auto-remove them from the other queue and notify (DM or channel ping).
- No auto-fallback/forced merging — a queue that can't fill just waits.
- On pop: all 6 players locked out of queueing until their series is reported/resolved.
- **Series timeout**: auto-cancelled (void, no MMR change) after 2 hours with no report (config: `series_timeout_hours`).
- **Mid-series abandonment**: detected by majority vote — **3 of the remaining 5 players** must each report the same player as abandoned (`/abandon`) before the series cancels immediately (void, frees the other 5 right away rather than waiting the full timer).
- **Implemented** (`web/lib/discord/abandon.ts`): `/abandon target:<@user>` infers the series from the channel (`id:` admin-gated override, mirroring `/report`/`/sub`); self-targeting is rejected in application code (not a DB constraint — see migration `0007_subs_abandon.sql`'s comment) since without it a target could vote for themself and short the "3 of the remaining 5" threshold. Votes upsert into `crl6mansqueuebot_abandon_votes` (PK `(series_id, voter_player_id)`, so re-running `/abandon` with a different target overwrites the voter's earlier pick rather than stacking). On the 3rd vote for the same target, settlement uses the same atomic `UPDATE...WHERE status='active'` claim pattern as `/report`, so a simultaneous `/report` and vote-threshold crossing can't double-settle; the losing side just gets "already settled" instead of a corrupted state. A settled void clears any pending `sub_requests`/`abandon_votes` (`clearPendingSeriesState`, `web/lib/discord/matchChannels.ts`) and posts a public "series cancelled" message before the same 30s-delayed channel deletion `/report` uses.

### Substitutes
- A player who needs to leave mid-series runs `/sub nominee:<@user>` inside their match channel (series is inferred from the channel; `id:` stays available as an optional override for admins running it elsewhere), **nominating a specific replacement**.
- The nominee must accept via a button before the swap happens.
- The sub inherits the leaving player's team seat and finishes the series; MMR (Rank Queue only) applies to whoever actually finishes, not the original player.
- Pending sub requests expire after a configurable timeout (`sub_request_timeout_minutes`) so a series can't hang forever on an unanswered nomination.
- **Implemented** (`web/lib/discord/sub.ts`, migration `0007_subs_abandon.sql`): `crl6mansqueuebot_sub_requests` follows the same "existence = pending" convention as `series_lobby`/`queue_members` — a row is deleted (not status-flagged) on accept, expiry, or series settlement, with the leaving player's id as part of the PK so they can't have two outstanding nominations at once. The nominee's own player row isn't created until they click Accept (`getOrCreatePlayer` at that point, using their own interaction payload for the display name) — this sidesteps parsing Discord's `resolved` payload for the USER-type `nominee` option, which the interaction's plain `options[].value` (the nominee's Discord id as a string) is enough for. Acceptance is an atomic delete-claim (`DELETE...WHERE series_id=... AND leaving_player_id=... RETURNING`, same pattern as `admin.ts`'s `removeAdminRole`) so a double-click can't double-accept. On accept: `series_players` swaps the leaving player's row for the nominee's (same team), the nominee is cross-removed from any queue they were sitting in, any `abandon_votes` referencing the leaving player (as voter or target) are deleted as stale, Discord channel permissions are swapped on both the text channel and the leaving player's team voice channel, and a best-effort `PATCH .../members/{id} {channel_id:null}` force-disconnects the leaving player from voice (permission revocation alone doesn't kick someone already connected). Before the swap, accept also re-checks `isPlayerLockedInActiveSeries` (exported from `queue.ts`, the same helper `/join` uses) on the nominee — without this, a player already locked into a different active series could accept a sub here and end up in two `series_players` rows at once, double-counting toward MMR; a rejected accept leaves the original player free to `/sub` again with someone else. Pending sub-request expiry is swept every minute alongside the other timeout checks (`web/app/api/discord/sweep/route.ts`), using the same atomic delete-claim so it can't race a last-second Accept click.

### Team formation (on pop)
- Players vote **Balanced** vs **Captains**; first option to reach 3 votes (of 6) wins.
- Players may set a personal **default vote preference** (`/vote-default`), auto-cast each pop, still overridable per-game.
- Exact 3-3 tie resolves to **Captains**.
- **Captains mode** (default): best two players by MMR become captains. "Worst two in lobby" exists as a config toggle, not default — scope is always within the 6-player lobby, never full-server MMR. Draft order for the remaining 4 players (decided this session): **Captain 1 picks one player, then Captain 2 picks two players, then the last remaining player auto-assigns to Captain 1's team** (no button click needed for that final auto-assign — only 4 total picks/choices happen, not an alternating snake).
- **Balanced mode**: brute-force all 10 unique 3v3 splits of the 6 players (C(6,3)/2), pick the split with the smallest MMR-average difference between teams. No snake draft — brute force is strictly better at this scale.
- **Implemented** (`web/lib/discord/teamFormation.ts`): captains-draft state lives directly on `series_lobby` rows (`is_captain`, `team`) rather than a separate draft-state table — whose turn it is is derived by counting how many non-captain rows already have a `team` set, not stored. Draft picks and vote casts both use an atomic `UPDATE ... WHERE column IS NULL` claim (relying on Postgres row locking) instead of an advisory lock, since each is a single-row race. One Discord message (`series.formation_message_id`) is created once and PATCH-edited through vote tally → draft turn UI → final teams-formed summary, mirroring the persistent-queue-message pattern from Phase 2.

### MMR / Elo
- Standard Elo, single rating per player, starts at **0** (can go negative — no floor/ceiling).
- Team rating = average of the 3 players' individual MMR.
- `E_A = 1 / (1 + 10^((R_B - R_A) / S))`, `R_new = R_old + K * (Score - E)`. Score = 1 win / 0 loss. Point change splits evenly across the 3 teammates.
- Defaults: `S = 400`, `K = 32` (both admin-configurable).
- **Provisional K**: elevated K-factor (config multiplier, default 1.75x) for a player's first N **Rank Queue** games (config `provisional_games`, default 10), then drops to standard K.
- No flat streak bonus anywhere. Rank Queue's incentive is that tight matchups already yield near-max K gains on wins; Universal Queue's incentive is fill speed/accessibility, not MMR (moot now since it's unranked). An optional cosmetic-only reward (role/flair) for Rank Queue volume is allowed later, kept fully outside the MMR formula.
- **Implemented** (`web/lib/mmr/elo.ts`, wired into `web/lib/discord/report.ts`): a `players.rank_games_played` counter (migration `0006_reporting_mmr.sql`) tracks the provisional-K window specifically — distinct from `total_games_played`, which counts both queues for placement bootstrap. When teammates have different provisional status on the same result, each player applies their *own* K to their third of the (Score − E) split rather than blending the three into one team K — the spec's "elevated K for a player's first N Rank Queue games" is a per-player property, so this isn't zero-sum across teams when K differs, which is expected. Each reported series' exact per-player delta is stored on `series_players.mmr_delta` (also added in `0006`) so a future `/admin unreport` (Phase 8) can unwind it exactly, without needing to reconstruct a reverse delta after later games have already moved everyone's rating.

### Bands / ranks
- 4 percentile-based bands (Mohs-hardness themed): **Iron → Garnet → Emerald → Sapphire**. **Prism** is a season-end-only Top 10 tier layered on top, not part of the daily band recompute.
- **Bands gate nothing — Rank Queue is open to everyone immediately, no placement requirement** (revised this session; supersedes the original placement-gated design below). There is a single shared `#rank-queue` channel, not one channel per band — the bot pools all Rank Queue joins together with no band segregation in matchmaking (same simple join-order pop as Universal Queue). The `Placed` role still exists and is still granted/revoked on band change, but it's now purely informational (confirms a player has a real band from the daily recompute) rather than an access gate (decided this session, see "Discord bot commands & channels" below).
- Cutoffs recomputed **daily** (not live per-game), over all **placed** players' MMR.
- **Placement**: a player is unbanded (`is_placed = false`, shows `NA` band/MMR on the leaderboard) until `total_games_played >= placement_games_required` (config, default 10; counts both queues — see "Placement bootstrap" below). Prevents a day-1 Sapphire off a lucky streak from zero games. No longer gates Rank Queue access (see above) — only controls whether they're included in the daily percentile pool and shown a real band.
- **Promotion**: immediate at daily recompute if MMR percentile crosses into a higher band. No safeguard needed.
- **Demotion**: two stacked safeguards, grace checked first —
  - **Grace period**: first **3** Rank Queue games played in a new band are demotion-immune regardless of standing (config `grace_games`; the original spec doc said 2 — superseded by the Q&A doc's answer of 3).
  - **Hysteresis buffer**: after grace expires, only demote if more than `hysteresis_pct` (default 5) percentile points below the promotion-in threshold.
- On any band change: swap the player's Discord role, DM them the change.

**Placement bootstrap (revised this session — no longer a chicken-and-egg problem):** originally Rank Queue required `is_placed = true`, which only Rank Queue itself could earn toward (since only Rank Queue moves MMR and only games-played unlocks placement) — that circularity was resolved by routing brand-new players through Universal Queue first. That gate is now removed: a brand-new player can jump straight into Rank Queue and start earning MMR (starting at 0) from game one. They still show as unbanded (`NA` band/MMR on the leaderboard, excluded from the daily percentile pool) until `total_games_played` crosses `placement_games_required`, at which point the next daily recompute assigns them a real band off whatever MMR they've accumulated by then, from either queue.

**Band percentile cutoffs (decided this session via user question, not specified in the original spec):** the original docs establish the *concept* of percentile-based bands but never pin exact split points. Skewed toward the lower bands rather than equal quartiles, per the user's choice: cumulative cutoffs `band_cutoff_garnet_pctile=40`, `band_cutoff_emerald_pctile=70`, `band_cutoff_sapphire_pctile=90` (all admin-configurable) — Iron is below the 40th percentile, Garnet 40th-70th, Emerald 70th-90th, Sapphire 90th and above.

**Implemented, Phase 5** (`web/lib/discord/bands.ts`, `web/app/api/discord/recompute-bands/route.ts`): `recomputeBands()` is triggered daily by `pg_cron`/`pg_net` (migration `0009_schedule_bands_cron.sql`, `0 9 * * *`, reusing the same Vault secret and `x-sweep-secret` header convention as the existing sweep job — no need for a second secret since both are pg_net-triggered background jobs with the same trust boundary). It pulls every non-test-data player who is either already placed or has just crossed `placement_games_required`, combines both groups into one MMR-sorted pool, sorts that pool by MMR (tied MMR broken by `total_games_played` descending — more games at the same rating is treated as a more established rank — then player id as a final deterministic tiebreak), and computes each player's percentile rank as `(index + 1) / n * 100` before assigning bands off the cutoffs above. Demotion applies grace first (`band_games_played < grace_games` is immune, short-circuiting before hysteresis is even checked) then hysteresis (only demote if the player's percentile is more than `hysteresis_pct` points below the *promotion-in threshold for their current band*, not the raw target-band cutoff — e.g. promoted into Sapphire at the 90th percentile, only demoted back out below the 85th at default settings). `band_games_played` (new column, migration `0008_bands.sql`) resets to 0 on every band change and increments alongside `rank_games_played` on every Rank Queue `/report` settlement (`report.ts`) — it's distinct from `rank_games_played` (lifetime Rank Queue count, powers provisional-K) and `total_games_played` (both queues, powers placement bootstrap). Role sync (`crl6mansqueuebot_band_roles`, admin-settable via `/setbandrole band:<Iron|Garnet|Emerald|Sapphire|Placed> role:<@role>`, mirroring `/setqueuechannel`'s upsert pattern) removes the old band's role and grants the new one, plus grants `'Placed'` once on initial placement only (never revoked on subsequent promotion/demotion) — purely informational since the Rank Queue access gate was removed (see "Bands / ranks" above). Since the recompute has no Discord interaction to pull `guild_id` from (unlike every other bot code path), a new `getGuildId()` helper in `rest.ts` auto-detects it the same way `register-commands.mjs` already does (`GET /users/@me/guilds`, throws if the bot isn't in exactly 1 guild unless `DISCORD_GUILD_ID` is set), cached at module scope for warm-instance reuse. Band-change DMs reuse the existing best-effort `sendDirectMessage` helper. Test-data players (dev panel) are excluded from the whole pool — their fake `discord_id`s aren't real guild members (role sync would 404) and would distort real percentile cutoffs if mixed in.

### Seasons
- Monthly, resets at the start of each calendar month.
- **Soft reset**, piecewise and continuous at 0 (revised this session — supersedes the original linear median-compression formula, which could *increase* a below-median player's MMR; every positive-MMR player now strictly loses MMR, and MMR can never cross below 0 as a result of decay):
  - `mmr >= 0`: `new_MMR = (mmr * median) / (median + decay_factor * mmr)` — a hyperbolic decay toward 0, scaled by the pool median; asymptotically safe (never negative for a non-negative input), and every player above 0 loses something (no flat dead-zone).
  - `mmr < 0`: `new_MMR = mmr / 3` — MMR has no floor during normal play (see "MMR / Elo"), so a negative-MMR player recovers 3x closer to 0 each season rather than jumping straight back.
  - Both pieces equal 0 at `mmr = 0`, so the combined function has no discontinuity, and both pieces are strictly increasing in `mmr`, so the combined function never reorders players by MMR (same invariant the original formula had — see the decay-order note under "Implemented" below).
  - `decay_factor` default `0.25` (config), same key as before.
- **Top 10 / Prism role**: calculated once at season close only (not daily). Requires **8+ games played** that season (config `top10_min_games`) to qualify; tiebreak at the #10 cutoff is most games played. Auto-assigned, stripped from previous holders.
- Leaderboard shows **live** season standings throughout the season, including a "currently projected Top 10" indicator — the role itself only changes hands at season close.
- **Implemented** (`web/lib/discord/seasonClose.ts`, called from `web/lib/discord/seasons.ts`'s `/newseason` when a current season exists): `/newseason`'s first step is now an atomic claim — `UPDATE seasons SET is_active=false WHERE is_active=true RETURNING` (same pattern as `report.ts`/`teamFormation.ts`'s settle claims) — so a double-fire (Discord retry, or an admin re-clicking because the deferred ACK gives no instant feedback) can't run the soft reset twice; a losing caller sees 0 rows back and just opens the next season with no close to report. `closeSeason()` then runs in a fixed order: (1) **standings** — every player with >=1 `reported` series in the closing season (both queue types combined, matching CLAUDE.md's "Queueing" note that Universal Queue games count toward general games-played counters) gets a `season_history` row, ranked by pre-decay MMR descending, ties broken by season games played descending then player id (same tiebreak philosophy the user established for the daily band recompute's MMR ties — see "Bands / ranks"); `season_rank` is written for *every* participant (matching the leaderboard's "based on final MMR ordering among all season participants" language — display-gating by `season_rank_display_min_games` happens separately, at read time, on the leaderboard itself), `made_top10` is `true` for the top 10 among the subset with `season_games_played >= top10_min_games`, with the #10 cutoff tiebreak (most games played) falling out of the same sort; (2) **Prism role sync** — reuses the existing `crl6mansqueuebot_band_roles` table/`/setbandrole` command rather than a dedicated table (migration `0010_prism_role.sql` extends the table's check constraint to accept `'Prism'` and adds `players.is_prism boolean` to track current holders without scanning live Discord role membership); `recomputeBands()` (`bands.ts`) never assigns or reads the `'Prism'` key, so it stays structurally separate from the 4 real bands despite the shared storage; strips the role + DMs anyone in the previous `is_prism=true` set who isn't in this season's new Top 10, grants + DMs anyone new; (3) **MMR soft reset** — the piecewise decay (see "Seasons" above) applies to *every* currently placed, non-test player, not just this season's participants (a player who sat out the whole season still decays). This intentionally runs *after* standings are written, using the pre-decay MMR values, so `mmr_at_close` reflects what players actually earned. No `recomputeBands()` call follows the decay — both pieces of the formula are strictly increasing in `mmr` and agree at the `mmr = 0` boundary, so the combined function is strictly increasing across its whole domain and can't reorder any player's MMR-sorted position, and therefore can't move a single percentile rank or band; the next daily cron tick handles ordinary band drift exactly as it would on any other day. Test-data players are excluded from every step (standings, Prism, decay), same treatment as `bands.ts`. This is manual-trigger only for now (an admin runs `/newseason`) — no scheduled monthly `pg_cron` rollover, deferred as a possible small follow-up. Every select that can grow with community/season size (series lookup, series-players participation count, participant player rows, the decay pool) is paged in `PAGE_SIZE=1000` batches via a shared `fetchAllPages()` helper rather than trusting a single unbounded `select` — PostgREST silently truncates results past a project-configured row cap with no error, which would otherwise corrupt `season_rank`/Top10/Prism at a large enough season with no visible symptom; `.in(id list)` calls are additionally chunked (`ID_CHUNK=200`) to keep request URLs bounded. Not fully atomic end-to-end: the `is_active` claim prevents a *double* close, but if `closeSeason()` throws partway through (e.g. mid Prism-role-sync or mid-decay), the season is already closed and there's no automatic re-entry or rollback — acceptable at this project's scale, but a future admin-tools pass could add a resumable/idempotent retry if that ever bites. This logic is build/lint-verified but not yet exercised against real data — `closeSeason()` excludes test-data players by design, so the dev panel's synthetic data can't drive a real dry run; the first real `/newseason` close is effectively the first live test.

### Reporting & disputes
- `/report result:<win|loss>` (and `/r result:<win|loss>` alias) — usable by any of the 6 participants from anywhere (no channel requirement). Specifies their team's outcome (win or loss). Series is inferred from the player's active series via `series_players`. Settles immediately on first report, no confirmation window.
- **Implemented** (`web/lib/discord/report.ts`): Series lookup finds any `active` series the player is locked into (via `isPlayerLockedInActiveSeries`). Settling is an atomic `UPDATE...WHERE status='active'` claim (same pattern as `teamFormation.ts`'s vote/draft resolution) so a race between two participants' `/report` calls can't double-settle. The deferred ephemeral response is deleted on success (no private feedback to the reporter), while error paths edit it with useful context ("You're not part of an active match", "already reported", etc.). The report summary is posted as an embed to the admin-specified report channel, splitting players into **Winners**/**Losers** sections; each line shows that player's MMR delta, resulting MMR, and current band (Rank Queue only — Universal Queue lines just note "no MMR change", test-match lines note "no stat changes"). Band is the player's last daily-cron-computed snapshot (`players.band`), not live-recomputed per game, so it may lag this exact result by up to a day. The deferred ephemeral reply to the reporter is deleted (leaving no private feedback) — the full summary in the report channel is the only output.
- `/admin unreport id:<series_id>` — reverses a report, unwinds MMR for all 6 affected players. Restricted to a configurable Discord role (not hardcoded user IDs).

### Admin toolset
- `/admin unreport`, `/admin cancel-series`, `/admin adjust-mmr`, `/admin force-leave`, `/admin recompute-bands`, `/admin config get|set`, `/admin audit-log`. (`/newseason` stays a standalone top-level command, not an `/admin` subcommand — see "Admin commands" below; the original spec's `/admin new-season` never got built as such.)
- All admin actions get a basic audit log entry (actor, action, target, details, timestamp).
- **Implemented** (`web/lib/discord/adminTools.ts`): four design questions were resolved before building — (1) `unreport` reverses a `'reported'` series to the existing `'void'` status rather than a new DB status, since the audit log entry already distinguishes an admin-corrected void from a genuine timeout/abandon void; the atomic claim is `UPDATE...WHERE status='reported'` (same claim pattern as everywhere else), and the unwind mirrors `report.ts`'s writes in reverse per `series_players` row — `total_games_played` always decrements, and for Rank Queue series `mmr -= mmr_delta`, `rank_games_played`, and `band_games_played` also decrement (each clamped at 0, in case a since-happened band change already reset `band_games_played`). A 6-mans series has 6 `series_players` rows, not 10 — an earlier draft of this bullet had the wrong count. (2) `adjust-mmr` takes a `target:<@user>` plus **either** `delta:` (relative) **or** `mmr:` (absolute), never both — validated as an XOR in application code, since the user wanted both input modes supported rather than picking one. (3) `force-leave target:<@user>` handles both possible states rather than assuming one: dequeues the player from any queue they're sitting in (reusing the now-exported `refreshQueueMessage` from `queue.ts`) *and* voids any series they're locked into (`series_lobby`/`series_players`, same lookup `isPlayerLockedInActiveSeries` uses but returning the full row) — an admin bypass for a stuck/unresponsive player when the other 5 can't gather the 3 `/abandon` votes. (4) `config get|set` validates its `key:` argument against `KNOWN_CONFIG_DEFAULTS` (`config.ts`), a hardcoded allowlist mirroring this file's own "Config values" table (and duplicated a third time as Discord command `choices` in `register-commands.mjs`, matching the existing duplication precedent `setbandrole`'s band choices already established) — `get` with no `key:` lists every known key with its current DB value or fallback default. `cancel-series [id]` and `force-leave`'s series-void case both reuse two helpers split out of what was one combined function in `matchChannels.ts`: `claimSeriesVoid()` (atomic claim + clear pending sub/abandon state + public closing message) and `closeMatchChannelsAfterDelay()` (the 30s wait + delete) — split apart specifically so the admin's ephemeral reply can fire right after the claim succeeds instead of blocking on the 30-second warning window, matching the reply-before-delay ordering `/report` and `/abandon` already use. `recompute-bands` is a thin wrapper around the existing `recomputeBands()` (`bands.ts`). Discord's `SUB_COMMAND_GROUP`/`SUB_COMMAND` nesting (needed for `config get`/`config set`) required a new `CommandOption` type (`types.ts`) supporting nested `options`, plus a small tree-walker (`resolveAdminSubcommandPath`) that tells a subcommand hop apart from a leaf param by whether `options` or `value` is present on the node.

## Discord bot commands & channels (resolved design, Phase 2 implementation starting)

Design discussion held this session, ahead of Phase 2 (queue system) implementation. This section originally described a button-driven queue UI, revised at the time in favor of buttons over slash commands to avoid channel spam. **Superseded in a later session**: queueing moved back to slash commands (`/q`/`/queue`, `/l`/`/leave`) so queue channels can stay plain text channels, using a delete-and-repost single-message pattern instead of buttons to still avoid clutter — see "Queue channels" below for the current design. The private-per-match-channel piece of the original design (for report/sub/abandon's implicit series context) is unchanged.

### Discord bot runtime architecture (decided this session)
- The bot is a Next.js API route, `web/app/api/discord/interactions/route.ts`, registered with Discord as the application's **Interactions Endpoint URL**. No `discord.js` `Client`/gateway connection, no privileged gateway intents (Server Members Intent etc. are gateway-only concepts and not needed) — member data arrives directly in the interaction payload.
- **Signature verification**: Discord signs every request with Ed25519; the route reads the **raw request body** (`await req.text()`, not `req.json()`) and verifies it with the `discord-interactions` package and the app's **Public Key** *before* parsing JSON — verifying after parsing is a known intermittent-failure trap. Unsigned/invalid requests get rejected; a bare `PING` gets an immediate `PONG` (Discord requires this to accept the endpoint URL when it's first registered).
- **3-second response limit**: Discord requires a response within 3 seconds, which heavier handlers (e.g. pop → create a category + 1 text channel + permission overwrites) can't reliably hit, especially through a cold start. Pattern: reply immediately with a **deferred ACK** (type 5, "thinking…"), then do the real work and edit the original response afterward via bot-token REST. Because a serverless function can freeze the instant it returns, the follow-up work is wrapped in **`waitUntil`** (`@vercel/functions`) so it keeps running after the ACK is sent — this round-trip needs to be verified working in production before the pop handler is built on top of it (see Phase 2 implementation tasks).
- **Queue status message updates** use bot-token REST, not interaction webhook tokens (those expire after 15 minutes and can't be used for later edits from an unrelated future join). **Revised in a later session**: rather than `PATCH`-editing one message in place, each join/leave now deletes the previously tracked message for that channel and posts a fresh one (`postFreshQueueMessage`, `queue.ts`), keeping exactly one live queue-status message per channel at all times — see "Queue channels" below. The channel→message ID mapping is still stored in the DB (`crl6mansqueuebot_queue_messages`) so the bot can find and delete the right message across restarts.
- **Background sweeps** (series timeout, vote timeout, sub-request timeout) have no interaction to hang off of, so they're driven by **Supabase `pg_cron` + `pg_net`**, calling a secret-guarded sweep API route on a `* * * * *` (every-minute) schedule — plenty of precision for timeouts measured in minutes/hours. This was chosen over Vercel Cron specifically to avoid depending on Vercel's cron-tier limits, since Supabase is already provisioned for this project.

### Queue channels
- **`#universal-queue`** — open to everyone, no role gate.
- **`#rank-queue`** — single shared channel, **open to everyone with no role gate** (revised — the `Placed` role no longer restricts access, see "Bands / ranks" above); the bot pools all joins together with no band segregation in matchmaking.
- **Slash commands, not buttons** (revised in a later session — queue channels stay plain text channels): `/q` or `/queue` to join, `/l` or `/leave` to leave, all channel-inferred (the bot looks up which queue a channel maps to via `crl6mansqueuebot_queue_messages`, populated by `/setqueuechannel`). Discord has no native command-aliasing, so `/q`/`/queue` and `/l`/`/leave` are 4 separately registered commands that resolve to the same two handlers.
- Each queue channel has exactly one **live queue-status message** showing the current member list. Instead of editing a message in place, every join/leave **deletes the previous tracked message and posts a fresh one** (`postFreshQueueMessage`, `web/lib/discord/queue.ts`) — chosen so the queue status always appears as the newest message in the channel (a plain text channel has ordinary chat happening in it too, unlike the old buttons-only design), while still keeping exactly one live status message and avoiding clutter.
- `/q`/`/l` reply **ephemerally** to the calling player only (confirmation or error — e.g. "already queued," "you're locked in an active series"); the shared status message is the public-facing state.
- Joining/leaving must be an atomic check-and-write (DB constraint/lock, via `join_queue`/`leave_queue` RPCs) so simultaneous joins at 5/6 can't over-pop a queue past 6.

### Voice channels (created per series on pop, in admin-specified category)
- On pop, the bot creates **two voice channels** in the admin-specified 6-mans call category (set via `/set6manscallcategory`), one per team, named after the teams and restricted so each team can only see/join their own channel.
- Visible only to the 6 players in the series. Admin-role holders get visibility via a **category-level** permission overwrite on the admin-specified category — it cascades to all child channels with no extra per-channel step.
- `/report`, `/sub`, `/abandon` are slash commands with no channel-inference requirement — players run them from anywhere (though `/sub` and `/abandon` typically run from queue channels for convenience). `/report` requires the player to specify `result:<win|loss>` to indicate their team's outcome.
- **Series end** (report settles it, abandon-vote passes, `series_timeout_hours` elapses, or admin `/admin cancel-series`) → voice channels are **deleted instantly** (no delay) → all 6 players unlocked to queue again.
- The existing 2-hour timeout sweep also checks for and cleans up **orphaned voice channels** (e.g. left behind by a bot crash/restart mid-series) in addition to force-cancelling the DB series record. **Implemented, Phase 4/6**: this backstop now also covers *reported* series — `/report`'s own instant deletion is a single in-function attempt with no retry, so the per-minute sweep additionally re-checks any series with `status='reported'` and `reported_at` more than 60 seconds ago that still has non-null voice-channel ids, and deletes+nulls them. This is belt-and-suspenders: normal reports are cleaned up by `/report` itself well before the sweep would ever see them.

### Team formation (voting in queue channel, drafting via DM)
- Bot posts a **vote message with two buttons** (Balanced / Captains) in the queue channel. First option to reach 3 of 6 votes wins; a 3-3 tie resolves to Captains (unchanged). Players' `/vote-default` preference auto-casts but is still overridable per game — players can click the other button to switch their vote.
- **Total vote silence** (nobody votes within `vote_timeout_seconds`) → **cancel the series** (void, same as an unreported timeout) rather than defaulting to a mode — decided this session.
- **Captains mode**: top two players by MMR become captains. Draft order: **Captain 1 picks one player → Captain 2 picks two players → the last remaining player auto-assigns to Captain 1's team** (3 button-driven picks total, then one automatic assignment — see "Team formation" under Game mechanics for the canonical rule). Pick buttons are DMed to whichever captain currently has the turn (`sendDirectMessage` with a `components` param, `web/lib/discord/teamFormation.ts`'s `sendDraftPickPrompt`) — no slash command needed, captains just click buttons in DMs. Discord's platform does not allow a bot to force-open a DM to a user who has DMs closed to non-friends/bots — this is a hard API limitation with no workaround, so on DM-send failure the bot falls back to posting the same pick buttons directly in the queue channel with a status line @-mentioning the captain and explaining their DMs are closed. Since a DM-originated button interaction carries no `guild_id` of its own, `processDraftPick` resolves it via `interaction.guild_id ?? (await getGuildId())`, reusing the same guild-auto-detection helper background jobs (e.g. the daily band recompute) already use.
- **Balanced mode**: no draft interaction — the bot computes the optimal 3v3 split immediately once the vote resolves and posts the teams-formed summary in the queue channel.

### Other user commands
- `/vote-default <balanced|captains>`
- `/rank` or `/profile` — shows MMR, band, placement progress, current queue status (works anywhere, not scoped to a channel)
- `/help` — public, lists available commands. Shows an additional "Admin commands" section only to callers who currently have admin access (see "Admin roles" below), since those commands would just error out for anyone else.

### Admin roles (implemented, Phase 2)
- Replaces the single `mod_role_id` config value (superseded — see "Config values" below) with an owner-managed multi-role system: any number of Discord roles can be granted admin access, not just one.
- `crl6mansqueuebot_admin_roles` table (`role_id`, `added_by`, `added_at`) holds the currently-granted roles. `crl6mansqueuebot_audit_log` table (`actor_discord_id`, `action`, `target`, `details`, `created_at`) records every admin action taken.
- Permission check is **owner OR has a role in `crl6mansqueuebot_admin_roles`** — checked via the interaction's `member.roles` payload (free, no extra REST call) plus a guild-owner lookup (`GET /guilds/{guild_id}`, comparing `owner_id`) as fallback. This makes "only the owner can use admin commands until a role is granted" fall out naturally with no special-casing, since the table starts empty.
- Admin-role holders get both (a) the ability to run admin commands and (b) visibility into all match categories/channels (see "Match channels" above).
- `/add-admin-role role:<@role>`, `/remove-admin-role role:<@role>`, `/list-admin-roles` — owner-or-admin-role gated, audit-logged. Note: any role with admin access can grant/revoke admin access to other roles (including itself) — this matches the literal request that admin roles can run admin commands, but is a privilege-escalation surface worth being aware of when deciding which roles to grant.

### Admin commands
All under `/admin`, gated by the admin-roles system above, all audit-logged: `unreport`, `cancel-series`, `adjust-mmr`, `force-leave`, `recompute-bands`, `config get|set`, `audit-log` — **implemented, Phase 8** (`web/lib/discord/adminTools.ts`, see "Admin toolset" above for the per-subcommand implementation notes). `cancel-series` with no `id:` resolves to whichever match the calling admin is currently sitting in (via membership, not channel — see "Queue channels" below); a standalone `/end` alias was considered and dropped as redundant with `cancel-series` covering the same no-`id:` case. `/setqueuechannel` (implemented, Phase 2 infra — run inside the target channel with `queue_type:<rank|universal>` to post/relocate that queue's persistent message there), `/newseason` (implemented, Phase 7 — atomically closes any currently active season, running the full close sequence in `seasonClose.ts` (standings, MMR soft reset, Top 10/Prism role sync — see "Seasons"), then opens the next one with `season_number` auto-incremented), and `/setbandrole` (implemented, Phase 5 infra, extended Phase 7 — `band:<Iron|Garnet|Emerald|Sapphire|Placed|Prism> role:<@role>` maps a band (or the Placed gate, or the season-end-only Prism Top 10 tier) to the Discord role the daily recompute or season close grants/revokes on change) are also gated this way, as standalone top-level commands rather than `/admin` subcommands — this was already true before Phase 8 and stays that way; the "Admin toolset" bullet list above no longer lists a `/admin new-season` subcommand to avoid the contradiction.

### Pop-to-report flow
6th join (via `/q`/`/queue`) triggers pop → bot cross-removes those 6 players from the other queue's status message → creates two voice channels in the admin-specified 6-mans call category (one per team, with permission overwrites restricting each team to their own channel) → posts vote message with buttons in the queue channel (or auto-resolves via `/vote-default` prefs) → team formation runs (draft via DM-driven picks with queue-channel fallback, or balanced split) → any participant runs `/report result:<win|loss>` in the queue channel or anywhere (no channel inference, uses `/report result:win` or `/report result:loss` instead) → match result posted to the admin-specified report channel (not the queue channel) → MMR updates (Rank Queue only) → band-change DMs fire if applicable → voice channels deleted instantly → all 6 unlocked to queue again.

## Leaderboard design (Phase 1)

Read-only, public, no auth. Three boards. A player only appears on any board once they've played at least 1 game (any queue).

### 1. Main Leaderboard (Rank)
- Single table showing every player with >=1 game played (placed or not) — sorted by band then MMR, with unplaced players (`is_placed = false`) grouped after all placed bands and ordered by total games played instead. This is Rank Queue standing only — Universal Queue never affects MMR, so it doesn't factor into this board's ranking.
- Columns: position, player name, band, current MMR, wins, losses, win rate, **last season rank**. Unplaced players show **NA** for band and MMR (revised this session — no separate "In Placement" section; originally spec'd as a split-out list, superseded in favor of one unified table).
- **Top 10 rows highlighted gold** — this is the *live projected* Prism cut, not the locked-in role (which only changes hands at season close). Purely visual on this board.
- **Last season rank**: the player's exact overall numeric standing at previous-season close (e.g. "#23"), based on final MMR ordering among all season participants. Shows **NA** if they played 9 or fewer games last season. Governed by a new config value `season_rank_display_min_games = 10`, kept separate from `top10_min_games = 8` (different thresholds for different purposes — this display cutoff vs. Prism-role eligibility).
- **Paginated**, 20 players per page, with a draggable range-slider control at the bottom to move between pages.

### 2. Season Stats Leaderboard
- Scope: current season by default, with a toggle to previous season (if one exists — only current + one prior, no deeper history for v1).
- **Queue-scope filter**: three-way toggle — All Queues / Rank only / Universal only — recomputes every stat column for the selected scope.
- Columns (sortable, so this board doubles as "most games played" / "best win rate" / etc. via column sort rather than separate pages): games played, wins, losses, win rate, longest win streak (that season, that scope), current streak.

### 3. All-Time Stats Leaderboard
- Same shape as the Season board (same queue-scope filter, same sortable columns) but lifetime totals instead of season-scoped. Longest win streak = all-time best; current streak = the same live current streak, still queue-scoped by the filter.

### New config values (add to the table above)
- `season_rank_display_min_games = 10`

## Website implementation notes

Vercel project (`crl6mans-queue-bot`) is **not connected to a Git repo** — `vercel deploy` without `--prod` still targets production (no branch to preview against), so any future CLI deploy from `web/` goes straight live unless a Git integration is connected first.

- No real game data source yet (bot doesn't exist). A password-gated dev panel at `/dev` (gated via `DEV_PANEL_PASSWORD` env var, stateless HMAC-signed session cookie) has "Add 10 test players" / "Remove all test players" buttons that generate/clear synthetic data (`is_test_data = true` on `players`/`series`, cascades on delete) for verifying the boards. Remove or re-gate this panel once the real bot is live.
- `DEV_PANEL_PASSWORD`'s value was exposed in an agent debugging session's terminal output — rotate it periodically since it's the only thing gating live write access to the seed/reset endpoints.
